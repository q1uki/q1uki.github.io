【Learning Notes】I2P Overview

Introduction

I2P (Invisible Internet Project) is a decentralized peer-to-peer (P2P) anonymous network, developed in JAVA. Similar to Tor, it encrypts and transmits data through several network nodes, but offers a higher degree of anonymity than Tor.
Tor primarily focuses on providing users with anonymous access to external internet services and operations, addressing “anonymity” and “hidden services”, with an emphasis on application layer design, building an “overlay network”.
I2P attempts to migrate existing internet services to the I2P network and provide service implementations within its framework, offering anonymous file sharing and anonymous web hosting, addressing “anonymity”, “hidden services”, and “security”, with an emphasis on network layer design, building a “virtual internet”.
(Reference: https://www.hanspub.org/journal/paperinformation?paperID=31299&btwaf=39457632)

Usage

  1. Installation
    JAVA version higher than 1.5 is required.
    Installation tutorials can be found on the official website: https://geti2p.net/en/download
    (Official note: Do not start i2p as root on Linux)

  2. Connecting to the I2P Network
    The first connection to the I2P network requires reseeding, which will be discussed in detail later.

  3. Proxy Settings
    To access websites via I2P, local proxy settings are required. The default HTTP proxy port is 4444, and HTTPS is 4445.

After successful startup, visiting 127.0.0.1:7657 will bring up the I2P console, including settings and various network metrics.

Mechanism

  1. Kad Algorithm
    Purpose: To disseminate and obtain network node information via seed nodes, and to break down raw transmission data into encrypted packets for scattered delivery through multiple cross-tunnels.

Features:
① Only a small fraction of the entire I2P network nodes are obtained.
② No directory servers are required.
③ Every host running I2P acts as a network relay, sharing bandwidth to forward data for others.
④ Once connected to I2P, it continuously acquires information about other nodes in the network, ensuring a sustained connection to the I2P network.

  1. Network Database (netdb)

  1. Garlic Routing
    When transmitting information, the information to be transmitted is first divided into multiple sub-messages, and then available intermediate nodes are randomly selected from the network database (netdb) to construct multiple encrypted tunnels.

    ① Tunnels
    When establishing a tunnel, the tunnel initiator must send requests and necessary configuration data to each hop selected by the initiator, and wait for confirmation from each hop before the tunnel can begin to be established. The tunnel establishment request sent by the initiator is encrypted, and only the tunnel creator can access the confirmation information from each hop. The maximum lifecycle of a tunnel is 10 minutes. Typically, a tunnel has 3 hops: a gateway node, intermediate nodes (adjustable quantity), and an endpoint. A complete unidirectional transmission includes an outbound tunnel and an inbound tunnel, involving at least 6 nodes.

    ② Peer Selection
    In selecting available intermediate nodes, I2P clients rely on previously monitored performance values and the current state of the network, without using active bandwidth probing. The I2P node selection algorithm can react very quickly to failed peers and other changes in network topology by continuously analyzing and ranking performance to select peers, rather than trusting claimed capacity.
    (Reference: http://www.3n1ac.com/index.php/archives/3/)

    ③ Tunnel Creation
    When a router node wants to create an outbound tunnel, it first checks its netDb folder to obtain a series of RouterInfo messages (which were obtained from real-time monitoring by the router), selects several nodes based on their performance and other metrics, strictly ranks these nodes according to a specific algorithm, configures the corresponding information for each selected node, and then forwards it.

    For an outbound tunnel, the tunnel creator is the tunnel gateway. It forwards the tunnel creation instructions and other pre-configured information to the next hop router—the tunnel’s intermediate node—in the previously selected and ranked order. The intermediate node, upon receiving the instructions, understands that a tunnel is to be created, continues to forward the instructions, and then replaces the request record with a reply record (indicating whether it agrees to participate in the tunnel), forwarding it to the next hop. When the message is forwarded all the way to the last node of the tunnel, i.e., the endpoint, it will forward the message to the corresponding inbound tunnel according to the previous instructions, and finally return it to the corresponding tunnel creator.

    Since tunnel construction incurs significant time costs and bandwidth load, and each tunnel has a maximum lifecycle of 10 minutes, after which it is destroyed, to ensure efficient service operation and provide redundant bandwidth for applications, each I2P router maintains a series of tunnel pools, where tunnels in each pool are used for the same communication purpose.

    ④ Endpoint Processing
    After the last hop in the tunnel receives and verifies the tunnel message, how the endpoint recovers the gateway’s data also depends on whether the tunnel is an inbound or an outbound tunnel.
    For an outbound tunnel, the endpoint, like other participants, encrypts and forwards the message using its layer key.
    For an inbound tunnel, the endpoint is also the tunnel creator, so it can iteratively decrypt the IV key and message using the encryption key and IV (Tunnel Initialization Vector) key for each layer, in the reverse order of the tunnel node routing.

I2P vs. Tor Comparison

  1. I2P Garlic Routing
    ① Completely decentralized, node information is stored in each I2P node, no directory servers needed.
    ② Multiple tunnels transmit received and sent information, and change every 10 minutes.
    ③ Transmitted information is split into multiple sub-messages and sent to the destination via different encrypted tunnels.
    ④ Slower speed.
  2. Tor Onion Routing
    ① Relies on directory servers to obtain relay nodes, making it susceptible to blocking.
    ② Single tunnel transmits received and sent information.
    ③ Faster than I2P.

Reseeding

Due to the Kad algorithm’s need for initial seeds to discover more node information, reseeding is required when connecting to the I2P network for the first time. This involves using a certain number of known active nodes to find other nodes.
Once connected to I2P, you can stay online indefinitely. If not used for a period (more than three months), reseeding might be needed again because current active I2P node information is lost.
I2P initially has built-in seed node information, but it might become invalid, requiring other reseeding methods.

  • Reseed Websites, Files
    I2P has several reseed websites built-in, which are the default reseeding method.
    You can also download reseed files from reseed websites; high timeliness is required.
    You can create and share reseed files with others (node information comes from your own network database).

  • Reseed Files
    Importing from files can be in zip or su3 format; su3 is a compressed file with signing capabilities.
    After unzipping, opening it with a notepad reveals I2P router information, including host, port, and key information for both NTCP and SSU protocols, which is what is displayed in the netdb of the console. The garbled part is the su3 file header information; a detailed table can be found in the official documentation.
    (Official documentation: https://geti2p.net/spec/updates#su3)
    (Based on incomplete observation, about 70 new nodes are added per hour)

Websites

I2P anonymous sites (eepsites) are websites within the I2P network. I2P site names end with .i2p and are found by identifying keys (i.e., encrypted base64 keys, which differ from common base64 by using ~ instead of / and - instead of +). Known site keys are stored in the local host.txt file.

  • Address Book
    I2P website keys can be accessed by subscribing to an address book, navigating websites, or importing addresses manually.
    For example:
    http://tino.i2p/hosts.txt
    http://stats.i2p/cgi-bin/newhosts.txt
    http://i2host.i2p/cgi-bin/i2hostetag
    http://no.i2p/export/alive-hosts.txt
    http://notbob.i2p/hosts.txt
    Taking notbob as an example, website categories include search engines, cryptocurrencies and trading, git, video, sqbl, etc.

  • Websites
    When visiting a website not in the address book for the first time, you can find the website’s key through a jump service.
