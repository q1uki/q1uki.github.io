Introduction

A Chinese-friendly reading note on the paper “LZR: Identifying Unexpected Internet Services”. It contains a large amount of machine translation and summarization of the original text, with my additions in italics.

Research Highlights

  • Investigated where services are deployed in the network and assessed the security posture of services deployed on unexpected ports.
  • Past scans assumed services were deployed on common ports, neglecting to scan unexpected service ports. Hosts responding to SYN scans did not complete the application-layer handshake, perhaps indicating hidden services or firewalls blocking further interaction.
  • By analyzing the behavior of uncompleted application-layer handshakes on unexpected service ports, firewalls and middleware were tracked to evaluate the effectiveness of anti-scanning measures.
  • Discovered that because services run on unexpected ports, L7 handshakes require pre-established secret protocol handshakes and data confirmation on each port. Services deployed on unexpected ports tend to have weaker security, such as some IoT devices.
  • Fingerprinted 88% of identifiable services with one packet, and identified 99% of identifiable unexpected services with five handshakes.
  • Sped up scanning by filtering out hosts that respond well, return SYN-ACK, but cannot complete the application-layer handshake.

Finding Real TCP Services

TCP activity does not necessarily indicate the presence of application-layer services due to security firewall tools like firewalls and middleware.

L4 and L7 Activity

Confirmed whether hosts responding to SYN-ACK packets completed the application-layer handshake for common port services. First, ZMap was used to send SYN packets, then ZGrab was used to attempt application-layer handshakes.

As a result, a significant portion of hosts responding to TCP did not complete the L7 handshake; completion was most likely on common ports and rare on uncommon ports.

The following analyzes the lack of application-layer response by examining state changes in the two scanning phases.

Connection Evasion

If firewalls indiscriminately spoof port-open packets, it would affect normal users. Therefore, firewalls need to judge whether to spoof responses according to certain rules, commonly by blacklisting IPs that send a large number of request packets.

During the TCP handshake with ZGrab, some ports did not respond to SYN-ACK, possibly due to network failure, DHCP interruption, or the scanner being blocked. Another unlinked host IP was used to scan a random ephemeral port to confirm connection evasion, revealing that 70% responded to this unlinked IP, meaning the previous IP was evaded. In the evaded IP blocks, 40% of hosts evaded /32 subnets, 10% evaded /24, and the largest was the Mexican ISP Alestra Net (ASN 11172), which evaded /20.

For IPs considered malicious, network hardware (e.g., routers) and host software (e.g., Snort) evade connections or dynamically block them. These methods prevent scanners from using a single IP to scan the network, forcing them to use multiple IPs to reach target hosts, thus increasing the difficulty for attackers. By comparing scan results from single source IPs and multiple source IPs, it was found that hosts that evaded connections did not provide legitimate services. If effective filtering can be applied during scanning, these can be safely ignored in research.

Do TCP-Responding Hosts Speak TCP?

During the second handshake with ZGrab, 96% of port services did not complete the application-layer handshake and returned SYN-ACK. Next, we explore whether hosts entered a state to exchange application-layer data or simply stopped responding after sending a SYN-ACK. The figure below illustrates how the scanner infers server states based on captured data. To enter the established state, the server only needs to send a SYN-ACK. Once the client sends an ACK, data can be transmitted normally, with the amount of data specified by the server’s window size in the SYN-ACK packet.

The server can respond with a zero-window in the SYN-ACK; when the server is ready to accept data, the client sends subsequent ACK packets to probe. ACKNOWLEDGE DATA in the figure is defined as the state where the server allows the client to send data and acknowledges the client’s data.

ACCEPT DATA captures whether the server is ready to receive data; servers reaching this state will keep the TCP connection open long enough to accept and acknowledge data. These two states are used to distinguish whether an established connection can exchange data.

A new scanner developed based on ZGrab establishes a TCP connection, sends two newlines, and infers the server’s TCP state.

After scanning random ports and IP addresses, it was found that 16% of services on common ports and 40% on rare ports could not enter the ACKNOWLEDGES DATA state.

Figure caption: (a) Percentage of ports with only SYN-ACK responses and those reaching the ACKNOWLEDGE DATA state among TCP-responding hosts; (b) Reasons why SYN-ACK returning hosts failed to enter the ACKNOWLEDGE DATA state.

Zero-Window DDoS Protection

Among services that do not accept data, 13% on common ports and 26% on rare ports return zero-window (nmap would consider a zero-window port as closed in a window scan) TCP packets and do not increase the window value to actively prevent clients from sending data. Among all scanned ports, 99.94% of hosts returning a zero-window do not increase the window value, 90% do not respond to secondary probes, and 10% reset the connection. These behaviors are network- or host-based, not service-based.

Among these, 99% of hosts that return a zero-window on one port will also return a zero-window on other ports of the same host. If a host never increases its window size in responses, then that host may never respond. Juniper’s Secure Services Gateway (SSG) proxies use this method to prevent network-based SYN cookie DDoS attacks. When a SYN-ACK packet with a zero-window is returned, after the client completes the three-way handshake with an ACK, the firewall sends a SYN packet to the server behind the firewall to establish a connection. During this time, it maintains a zero-sized TCP window with the client, and the middleware prevents the client from sending data to the server behind the firewall.

Zero-window SYN-ACK is deployed across entire subnets, with 90% of IPs exhibiting zero-window SYN-ACK within subnets larger than /24. The Florida Department of Management Services issued 16% of all zero-window SYN-ACKs across its entire network and accounted for 3% of all SYN-ACKs on random ports. Compared to subsequent RSTs, the TTL of SYN-ACKs was consistently near one hop, indicating they were issued by network devices.

Connection Interruption During Handshake

In addition to specifying a zero-window, 2% of hosts on each port never acknowledge data, despite the client returning an ACK, and cannot complete the three-way handshake. After retransmitting 7-8 SYN-ACKs, it is inferred that the server will not reach the established state. This is not just a TCP stack issue; under MCI Communication Services, IPs retransmitting SYN-ACKs on port 4567 exhibit the same behavior on other ports. The TTL of responses from real services is more than twice that of retransmitted SYN-ACKs, indicating selective packet dropping by middleware.

Interrupting connections during the handshake is a defensive action, a way for ISPs to protect users. CenturyLink (AS 209), Frontier Communications (AS 5650), and MCI Communications Services (AS 701) drop inbound traffic to 4567/TRAM post-SYN ports. Korea Telecom (AS 4766) and Axtel (AS 6503) account for 73% of connections interrupted on 7547/CWMP ports. This situation rarely occurs on common ports, with only 5% of TCP-responding hosts not accepting data and interrupting connections on port 80.

Resetting Connections

73% of common ports and 34% of rare ports that do not receive data and cannot reach the ESTABLISH state reset the connection immediately after the client completes the three-way handshake. According to RFC793, if a server does not wish to communicate with a client, it should close the TCP connection after the client acknowledges the SYN-ACK. This is how DenyHosts firewall responds to scanners. Blocking occurs more frequently on hosts running SSH than HTTP. Similar to networks sending zero-windows, this behavior is also caused by DDoS protection.

To prevent SYN flood attacks, middleware relays a SYN-ACK on behalf of the server, then establishes a connection with the server after the client completes the three-way handshake. If the server refuses the connection, the middleware terminates the client’s connection. Cisco IOS-based routers use this solution.

The likelihood of a host closing connections on common ports increases by 20%, as standard firewall policies accompanying Google load balancers accept communication on these ports by default to perform service health checks and rely on backend virtual machines to reset connections when ports are closed.

Dynamic Blocking After Handshake

Many hosts do not acknowledge any data at all and do not respond to subsequent handshakes.
To distinguish between hosts that evade the scanner after the handshake and those that simply do not acknowledge data, we simultaneously used the source IP of the initial connection and a new IP to perform an L7 handshake with hosts that initially responded but did not acknowledge data. 98% of hosts responded to the new IP, indicating they blocked the previous incoming connection. This dynamic blocking behavior only occurs after the three-way handshake, unlike the connection evasion in 2.2. The largest network with dynamic blocking after handshake is Coming ABCDE HK (AS 133201), accounting for 48% of all dynamically blocked IPs after handshake.

Other hosts that could not acknowledge data did not perform dynamic blocking because they never responded to anything after the actual handshake but always responded to all scans. Vodafone (AS 133612) and Webclassit (AS 34358) exhibited this behavior on all scanned ports, accounting for 66% of all IPs. Furthermore, TTL values did not match, indicating the presence of middleware.

Middleware Protection Capabilities

Dynamic blocking, i.e., completing the handshake without acknowledging data, is the most effective way to reduce scanning speed. This technique causes scanners to timeout by not receiving the scanner’s ACK. Zero-window SYN-ACK also causes timeouts, but zero-windows are easily filtered. Closing connections after the handshake only causes negligible slowdown.

Summary

SYN-ACK does not indicate the presence of a service. 16% of common ports and 40% of rare ports are protected by middleware and do not accept data. Below, we investigate why hosts using TCP do not always complete the L7 handshake.

Application-Layer Service Deployment

The previous section investigated L4-responding services that do not communicate with L7 services due to DoS and scanning protection. After excluding 28% of spoof services (see 3.1 for exclusion method), it was found that 27% of services on common ports and 63% of services on rare ports were not running the expected application-layer protocol. This section analyzes services that complete unexpected application-layer handshakes or acknowledge data but are not any identifiable application-layer protocol. Identifiable services deployed on unexpected ports constitute a large proportion, while only a small fraction are on IANA-assigned ports, with only 6.4% of TLS services running on port 443.

Finding Unexpected Services

When scanning a subset of IPs, for each responding service, an L7 handshake is first attempted with the expected protocol. If it fails, subsequent handshakes are performed using 30 different protocol scanners. 98% of IPs provide fewer than 4 unidentifiable services, while 2% of IPs provide unidentifiable services on over 60,000 ports. On rare ports, approximately 75% of unidentifiable services are provided by hosts that have unidentifiable services on almost every port. These hosts have unidentifiable services on most ports because some networks drop all traffic to security-sensitive ports. Among the top 50 networks with the most SYN-ACK responses on all ports, 28% dropped all traffic to port 445, and 10% dropped to port 23. Hosts with unidentifiable services on almost every port are concentrated in a few networks.

Additionally, in F5 Big-IP firewalls based on RST fingerprints, IPs respond on every port due to wildcard configurations when setting up the firewall or SYN-cookie cache overload. These hosts can be identified and excluded by checking if they acknowledge data on 5 random ports. After filtering, an average of 10% of common services and 25% of rare services remain unidentifiable (i.e., did not respond to any of the 30 handshakes).

Characterizing Unexpected Services

Common ports typically run expected protocol services; on port 80, most unexpected services are TLS-based.

Security of Unexpected Services

Many IoT devices occupy non-standard port 45, but research often only focuses on standard ports (14, 20, 27, 55, 62, 71). By manually identifying server certificates belonging to IoT manufacturers, we found that IoT interfaces are widely present on unexpected ports; 50% of TLS server certificates on unexpected ports belong to IoT devices, and the likelihood of these TLS certificates belonging to IoT devices is 5 times higher than on port 443.

For example, 35% of 8000/TLS devices in Korea Telecom are icctv devices (i.e., surveillance cameras), and 38% of 80/TLS devices are Huawei network nodes, distributed across 1% of international networks. In Korean networks, approximately 5% of TLS on port 8443 belongs to Android TVs, and at least 20% belongs to routers.

Unassigned ports also contain more TCP/UPnP devices. For example, the number of TCP/UPnP devices on port 49152 (mainly in Latin America and Asian telecommunications) is 12 times that on port 80, and on ports 58000 and 30005, it is 2 times that on port 80.

TLS services on unassigned ports are 1.71 times more likely to have certificates with known private keys than those on assigned ports. When scanning unassigned ports, we found more than twice the number of certificates with known private keys reported in previous work [32, 36]. For example, 40.2% of TLS hosts on port 8081 are DOCSIS 3.1 wireless gateways in Argentine Telecom (AS 10481 and 10318) using the same OpenSSL test certificate and known private key, and 39% of TLS hosts on port 58000 are Qno wireless devices using the same self-signed certificate and known private key. On 23% of scanned ports, public keys are more often shared (up to 1.7 times) than on port 443 (e.g., 80/TLS is 1.5 times more likely to be shared). Nevertheless, previous work on encrypted keys on the Internet limited analysis to 443/HTTPS, 22/SSH, 995/POP3, 993/IMAPS, and 25/SMTPS.

Summary

Many popular protocols are distributed across all 65K ports, not just IANA-assigned ports.

Efficient Service Identification

Previously, for L7 handshakes, 30 different handshakes (probes) were sent. This section discusses sending five handshake messages to discover 99% of identifiable unexpected services.

Protocol Discovery

Two directions for accelerating protocol discovery:

  1. Methods to trigger protocol identification responses across a large number of protocols.
  2. Optimizing handshake order for efficient service discovery.
    • Banner fingerprinting.
    • Optimal handshake order.
      On IANA-assigned and ephemeral ports, simply opening a connection and waiting can identify over half of unexpected services. For IANA-assigned ports, waiting and sending a TLS Client Hello can discover 80% of unexpected services. On ephemeral ports, waiting and HTTP discover 83.4% of services.

Impact of L7 Filtering (Application-Layer Fingerprinting)

One reason for not identifying all services is that even if the protocol guess is correct, the chosen handshake parameters might be rejected. (e.g., in SNMP, the server first acknowledges data, then sends a TCP RST to reject requests where the correct community string is not specified in the first data packet). Will this L7 filtering strategy prevent the server from sending any data to the client, hindering fingerprinting efforts?

Experiment: Taking PPTP as an example, according to RFC2637, add a magic cookie option (a pending constant for synchronizing TCP data streams), setting it to 0x1A2B3C4D in one request and 0x11111111 in another. The standard stipulates that if synchronization is lost, the TCP session for the control connection must be closed. Therefore, if the magic cookie is incorrect, the number of IPs sending data to the client will decrease. The experimental results are shown in the figure below, indicating that handshake parameters indeed affect the number of IP responses.

Responsive IPs acknowledge data and close the connection upon receiving a GET request, preventing the scanner from identifying the service as HTTP. By sending a request with options, 72% of IPs responded with 501 (Method Not Implemented), and 17% of IPs responded with 405 (Method Not Allowed), confirming they use HTTP. IPs specifically responding to options requests are not limited to a particular network, being present across 5.3% of ASNs. However, the difference in responses to requests with and without options is smaller for port 80.

For TLS, according to RFC8446, a handshake failure should generate an error message and notify the application before closing the connection. However, when an incompatible cipher is provided, 2.65% of IPs will simply close the connection without any application-layer errors. In PPTP, for example, an incorrect magic cookie will cause 67.1% of IPs to not respond with any data. The host performs application-layer filtering based on options and does not send any data to the client. For scanners, the challenge lies in guessing the correct parameters to elicit a proper response from the host.

Impact of Handshake Order (Corresponding to Optimal Handshake Order)

Similar to how handshake options can prevent server responses, repeatedly attempting incorrect handshakes before the correct one might prevent service identification. To assess whether hosts filter or reject connections after receiving incorrect L7 information, the following experiment was used: send consecutive HTTP GET and TLS hello messages to all IANA-assigned ports in 1% of IPv4, and compare the number of hosts that complete subsequent handshakes after sending expected L7 data versus after sending incorrect L7 data.

Across different protocols, sending incorrect L7 data leads to 30% of subsequent handshakes failing compared to hosts found when scanning the protocol directly. Some hosts, even if fingerprinting is successful, cannot complete subsequent handshakes. For example, sending non-Telnet data to a Telnet server causes 17% of subsequent handshakes to fail, with 65% returning TCP RSTs and 35% not returning SYN-ACKs. Sending an HTTP GET to a TLS server causes 29% of subsequent TLS handshakes to fail. This behavior is similar to Cisco IOS’s Login Block feature, which allows administrators to temporarily block connections to L7 services after login failures.

This phenomenon often occurs after the host sends protocol identification data, possibly because this is the first time the host stores application-layer state information about the server-side of the connection. It does not prevent fingerprinting, but only blocks subsequent handshakes after the identified data has been sent to the scanner. Waiting for a period after fingerprinting before performing subsequent handshakes can significantly increase the handshake success rate; 99% of hosts can complete subsequent handshakes within 2 minutes after being fingerprinted.

Sending incorrect handshake data can prevent some services from completing subsequent handshakes, even if fingerprinting is successful.

Summary

L7 scanning is limited by services requiring specific handshake options to respond. Results show that many hosts respond to incorrect L7 handshakes by sending data that helps identify the service. A single HTTP GET request can detect 16 out of 30 protocols, and 99% of unexpected services are identified within 5 handshakes (referring to five different types of handshakes). This forms the basis for building LZR.

LZR

Introduces the LZR system, a scanner that efficiently identifies Internet services. It can be used with ZMap or as a plugin between ZMap and an application-layer scanner (ZGrab) to indicate what subsequent handshakes to perform.

LZR’s advantages lie in its “fail-fast” method during scanning and its “fingerprint-everything” method for protocol identification, primarily relying on the following ideas:

  • Ignoring non-acknowledging hosts:
    40% of services sending SYN-ACK never reach the acknowledge data state. These services cannot complete an L7 handshake and can be ignored during scanning. Non-acknowledging services force stateful scanners to open operating system sockets and wait for timeouts. By sending an ACK application-layer packet with a TCP handshake, non-acknowledging hosts are filtered out, similar to ZMap’s stateless SYN scan.

    Attached are the principles of two port scanning prevention tools:
    1. Use a dynamic service signature database to generate fake banners, dynamically generating valid service signatures based on service signature regular expression databases to respond to service probes (e.g., nmap -F -sV).

    The only way to identify if a service is being emulated is through protocol probes.

    2. Preventing port scanning using TCP protocol

    My summary for determining non-acknowledging state:
    1. If an IP responds with SYN-ACK, but even one port returns a zero-window, it is highly likely that other ports will also return a zero-window. This indicates a protection policy.

    2. If SYN-ACK is returned but subsequently sent data is not acknowledged, it is middleware (firewall) sending the ACK on behalf of the server, not providing a real service.

  • Multi-listen.
    Up to 96% of services on each port run an unexpected protocol. Of the 30 protocols scanned, 8 protocols involve the server sending data first, and 10 protocols send fingerprint data when an incorrect L7 handshake is sent.

    By always waiting and then fingerprinting invalid server responses, we can identify 16 out of 30 protocols by sending a single packet. The scanner only needs to perform minimal computation to fingerprint the service: the first packet from the server identifies the running protocol, which does not require a full TCP/IP stack.

Scanning Algorithm

LZR receives a stream of SYN-ACK packets from ZMap or (IP, port) tuples for scanning. After obtaining complete connection details, LZR first filters out hosts that send with a zero-window (based on previous research, it was found that among hosts responding with SYN-ACK, 99% of those returning a zero-window on one port will also return a zero-window on other ports). For non-zero-window hosts, LZR will send an ACK packet containing the first packet handshake data of the expected protocol to continue the connection. If any type of data is received as a response, it will fingerprint the data and close the connection.

If the host neither acknowledges data nor closes the connection, LZR will use the PUSH flag and retransmit the data (see 5.3 for details). If the host does not acknowledge data (e.g., never responds or resets the connection with RST without acknowledging), LZR will mark the host as potentially not hosting a real service and will not continue with further connection attempts.

I deployed a simple rule to return a SYN-ACK packet upon receiving a SYN packet. When using nmap service detection, the PSH flag was also set to 1. After sending 5 times, the connection was disconnected, and a new connection was opened to send the next probe. This was time-consuming, and the final result was ‘open’, with the provided service uncertain, marked with a question mark.

If the host acknowledges data but sends no data in response (i.e., the server does not respond or closes the connection immediately thereafter), LZR closes the connection, starts a new connection, and sends the next handshake.

This process continues until the running protocol is identified or no other handshakes can be attempted. LZR can also selectively filter IPs that respond on almost every port (see Finding Unexpected Services) by simultaneously sending SYN packets to a user-specified number of random ephemeral ports and checking for SYN-ACKs.

Architecture

LZR is developed in Go language, provides all protocol handshakes in Appendix A, and uses libpcap to send and receive raw Ethernet packets. It can efficiently fingerprint services, using a single socket during scanning, adopting and continuing connections initiated by stateless scanners, and only needs to send and receive a single packet to complete service fingerprinting, without requiring a full TCP stack.

LZR takes a list of command-line parameters for the protocols to be tested and a SYN-ACK stream from ZMap or IP/port as input for scanning. A portion of the routines send subsequent ACK packets containing handshake messages and fingerprint the responses. It is easy to add new protocols/handshakes to LZR; each handshake generates a handshake interface that specifies the data to attach to the ACK packet and what to search for in the response packet to fingerprint the protocol.

Once fingerprint data is received, it first checks if the data matches the fingerprint of the attempted protocol (specified in the handshake interface). If not, it checks if any remaining fingerprints match. When using LZR, iptables rules must be set to prevent the Linux kernel from sending RST packets in response to SYN-ACKs it receives. Otherwise, LZR cannot adopt and continue these connections.

Evaluation

Performance Evaluation

  1. Compare ZMap and LZR in identifying services running on ports found by ZMap.
  2. Use LZR as a plugin between ZMap and ZGrab, instructing ZGrab on how to complete L7 handshakes for hosts found by ZMap. LZR can filter hosts faster than ZMap, especially advantageous on low-hit-rate ephemeral ports.
  3. ZMap/LZR+ZGrab offline scanning. First, we use ZMap and LZR to identify Internet hosts using known protocols and exclude this phase from our benchmarks. Then, in the second phase, we allow ZGrab to process services at full speed.
  4. ZMap+LZR offline scanning. First, we use ZMap to find candidate services and exclude this phase from our benchmarks. In the second phase, we tested how fast LZR could run full-speed fingerprinting services.
    We calculate the runtime performance of ZGrab and LZR using CPU cycles per second because both tools are CPU-bound: ZGrab completing full handshakes (e.g., encryption/decryption for TLS) and LZR’s fingerprinting (e.g., pattern matching) create the largest performance bottlenecks for each tool.

Number of Unexpected Services Discovered

Unexpected services can be identified while filtering out unresponsive services by analyzing responses to data contained in ACK packets. LZR uses a keyword fingerprinting strategy, identifying an average of 12 unexpected unique protocols across ports using only 1-2 expected handshakes; with just one expected handshake, it can discover 1.3 million IPs hosting 16 unexpected protocols on port 443 and 238,000 IPs hosting 18 unexpected protocols on port 80.

Host Filtering

LZR’s host filtering is similar to using only ZMap and ZGrab.

Detection Speed

LZR can identify services without completing an L7 handshake.
For many services that return a zero-window or do not acknowledge data, LZR detects services significantly faster than ZGrab.

Bandwidth Savings

Reasons:
(1) LZR does not attempt to retransmit ACKs to zero-window hosts to check for window size increases.
(2) LZR does not need to complete a full L7 handshake. However, when scanning with ZGrab+LZR, when most TCP-responding hosts serve ports with expected protocols, there is also an overhead in the total number of packets sent because LZR sends at least one additional ACK to obtain a fingerprint before retrying the handshake (e.g., for 80/HTTP, LZR+ZGrab sends 28% more packets in total than ZMap+ZGrab, even though LZR+ZGrab runs 1.2 times faster than ZMap+ZGrab).

Conclusion

A significant portion of hosts responding to SYN scans never complete the application-layer handshake, with most replies being handled by middleware in the absence of a real service. Many services are also found on unexpected ports, and compared to services on standard ports, those on unexpected ports tend to have weaker security.

LZR reduces the time required to perform application-layer scans on rare ports and identifies unexpected services running on ports. It identifies 16 protocols and 88% of identifiable services (referring to services identifiable by this system) with one packet, and 99% of identifiable services with five handshakes (sending up to 5 types of handshake packets).

Merely using the top N most popular ports is insufficient to assess which ports are most likely to host specific services, as HTTP and TLS are more widely used than most protocols.
